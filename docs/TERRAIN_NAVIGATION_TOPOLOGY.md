# 地形与高层寻路拓扑

这部分解决的是“地图里有哪些真正连通的区域，跨高度必须经过哪些坡道”，不是重新实现底层网格寻路，也不负责用地形逻辑掩盖单位局部避让问题。

## 数据来源

拓扑只能由一对匹配的数据生成：

- `TerrainMapSnapshot`：高度层、坡道方向和地面规则。
- `ClearanceBakeSnapshot`：静态障碍和 Small / Medium / Large 三档单位半径的实际可站立采样。

两者边界或来源地形哈希不一致时立即拒绝。生成结果记录来源地形、Clearance Bake 和自身稳定哈希，但它仍然是派生数据；真正移动是否合法继续由 `StaticWorld`、地形查询和 GridPathProvider 决定。

地形快照始终保持不变。建筑、岩石等临时物体不会回写高度格或坡道格，而是进入 `DynamicOccupancyGrid`；运行时拓扑另外记录它对应的动态占用版本。

## 连通区域

每种单位尺寸独立提取区域：

1. 读取该尺寸的 Clearance Bake 可走采样。
2. 排除坡道内部采样。
3. 只连接相同悬崖高度的相邻采样；对角连接要求两个正交邻格同时合法，不能从墙角穿过去。
4. 静态障碍已经进入 Bake，因此同一高度但被墙完整隔开的空间不会错误合成一个区域。

坡道采样被故意排除。如果直接使用 Bake 原始 Component，坡道会把上下层提前合并，系统就无法知道路线究竟跨过了哪一个坡道。

## 坡道口和宽度

连续、方向相同、低层高度相同的坡道格合并为一个坡道。每个坡道产生：

- 稳定的密集 ID。
- 低层入口和高层出口，位置来自相邻非坡道格的几何中心平均值。
- 原始物理宽度，即连续坡道格数量乘以地形格大小。
- 每种单位尺寸是否能通过；必须同时满足宽度要求、坡道内部存在该档 Bake 可走采样、两端能找到对应高度区域。
- 坡道及两侧入口会在局部 Bake 采样上真正连通；中间横跨整个坡道的静态岩石会让该坡道三档全部不可用，而不是因为附近还存在一个可走采样就误判。
- 交给 `ChokeController` 的接近距离和真实宽度。

宽度不是“能走/不能走”的布尔标签。20px 的测试坡道允许 Small、Medium，通过 Large 时明确没有高层路线；160px 坡道三档均可通过，并能据此计算安全车道容量。

## 高层路线如何工作

`TerrainTopologyRoutePlanner` 只计算坡道口序列：

- 起点和终点在同一实际区域时返回空高层路线，让 GridPathProvider 直接处理区域内绕障。
- 不同区域之间，以同一区域内的坡道端点和跨坡道连接组成稳定图，选择总代价较低的端点序列。
- 输出的 Waypoint 是真实坡道入口/出口，Choke ID 与拓扑坡道 ID 一致。
- 每个 Waypoint 之间仍调用生产 GridPathProvider，所以区域形状、建筑和静态障碍不靠直线猜测。

多层路线会依次输出两个或更多坡道。旧控制器只记住首个 Choke；现在单位离开前一个坡道后，会在进入下一坡道接近范围时根据当前身体、最终目标和坡道几何重新成组分配车道。重新接管还要求现有高层 Waypoint 同时包含该 Choke 两个端点，不能仅凭“当前位置和最终目标看起来跨过一条线”猜测。现有持久化状态足够描述当前 Choke，不需要新增另一份路线队列或升级回放格式。

## 车道边缘安全

理论上把单位中心放在“坡道半宽减单位半径”处可以通过，但邻居挤压会把最外侧单位推过悬崖边。车道容量因此额外保留 3px 挤压安全边，并且单位数量少于容量时只使用居中的必要车道，不把三个单位分散到五条理论车道的最外侧。

专项真实运行曾稳定复现一个单位在 96px 坡道边缘被推出约 3px 后长期卡住；修正后，同一三单位、两坡道、三高度层场景全部到达，两个 Choke 都在逐 Tick 状态中实际出现。

全量回归还捕获过一次错误接管：旧测试中的 Hold 堵口单位从狭口内部解除后移走，它没有高层坡道路线，却一度被几何扫描当成后续坡道成员，导致反向队列只到达 8/12。增加“两端点必须属于现有高层路线”的条件后恢复为双向 12/12、冲突 0；测试阈值没有放宽。

## 建筑落地后如何更新

`DynamicTerrainTopologyRoutePlanner` 在静态拓扑之上维护 Small / Medium / Large 三份运行时连通快照：

- 施工绿框、放置可行性查询和未来队列中的软 Reservation 不阻挡移动，也不改变拓扑。
- 建筑真正 Hard Commit 后，`RtsSimulation` 把准确的 footprint 矩形通知高层规划器；拆除、取消已落地建筑或建筑被摧毁时发送同一个移除矩形。
- 每种尺寸只重新查询“footprint 加该尺寸导航半径”所碰到的 Clearance Chunk。Chunk 外复用上一个版本的可走位图，然后全局稳定重标区域 ID 和坡道连接，确保结果与完整重建一致。
- 连续版本正常走增量；如果热恢复或外部调用一次跨过多个动态版本，则明确退回完整分析，不拿不完整的变化历史猜测。
- 热快照恢复会显式重建派生拓扑，即使恢复前后动态版本号碰巧相同，也不会复用另一个世界状态的缓存。
- 路线规划读取最新派生快照。近坡被建筑完整封住时，该坡道的尺寸掩码立即消失，平行路线改走另一坡；拆除后同一路线恢复。

双坡道专项覆盖 32×32、64×48、112×80、160×120 四档真实建筑：Small/Medium 只占局部空间时仍走近坡，Large/Huge 完整封住入口时改走远坡，移除后恢复。预览阶段动态版本保持不变。单次 Large 更新三档合计重采样 3,072 / 13,824 个采样（22.22%）、4 个 Chunk，普通放置/拆除完整分析次数为 0；另用“有建筑/无建筑”两份热状态反向恢复，路线准确恢复为远坡/近坡。
