# 终点协作收敛 V2

更新日期：2026-07-14

## 1. 目标问题

V1 已经提供唯一目标槽位、Hungarian 初始分配和两单位局部换槽，但无法保证解决下面的拓扑死锁：

```text
外围槽位先被占据
    ↓
形成稳定或缓慢微调的单位环
    ↓
后续单位持有内部预留槽位
    ↓
单位存在微小进展或反复短暂 Arrived，但长期无法稳定就位
```

这类问题不能只依赖普通 Steering 或“完全零进展”检测。单位可能持续以每秒一两像素蠕动、被推入 Arrived 后又被推出，导致通用恢复不断重置。

## 2. 第一层：驻留历史与 Overflow 兜底

### 独立终点历史

每个 32 人以上移动组维护：

- `DestinationBestDistances`：当前终点阶段观察到的最佳距离。
- `DestinationStallTicks`：未改善至少 3px 的连续 Tick。
- `DestinationNearTicks`：进入目标区域后的累计驻留 Tick。
- `DestinationOverflowed`：是否已经获得 Overflow 槽位。

局部两单位换槽只重置短期停滞，不清除累计驻留历史。`WaitingForPath` 和短暂 Arrived 也不会错误清空历史。

### 触发条件

- 只处理 32 人以上编队。
- 第一次进入距离自身槽位 180px 的区域后开始跟踪。
- 已进入后使用 260px 退出滞回，避免碰撞推挤导致反复进出。
- 终点阻挡先交给主动 Yielding；只有总驻留达到 900 Tick 后才允许新 Overflow，给临时让路留下处理窗口。
- 每 30 Tick 检查一次，每次最多处理 2 个单位。

“已就位阻挡者”包括严格 Arrived，以及距离自身槽位 18px 内且没有等待路径的单位。

### Overflow 槽位

Overflow 半径按同一目标附近的全部活跃预留人口计算，而不是只看当前命令波次：

```text
spacing = largest radius * 2 + 4
core radius = spacing * (sqrt(population) * 0.62 + 1.5)
```

从外侧最多 6 圈确定性采样候选位置。候选必须满足：

- 位于世界内且不碰静态/动态障碍。
- 从当前点到候选点的静态路径可通行。
- 不与任何单位的预留槽位重叠。
- 不与任何单位当前占用重叠。
- 相同输入按 Unit ID、Group ID 和固定角度得到相同结果。

获得 Overflow 后，单位退出 V1 换槽集合，避免重新被交换回内部槽位。

## 3. 第二层：主动 Yielding

第二层不直接改变内部受阻单位的槽位，而是让其路径上的已就位单位暂时离开。每个让路者使用显式状态机：

```text
None
  → MovingAside（前往临时让路点）
  → Waiting（保持通道打开）
  → Returning（返回原槽位）
  → None
```

### 3.1 候选与触发

- 只处理 32 人以上编队和仍持有内部目标的 Moving 单位。
- 受阻单位连续停滞至少 120 Tick，或在目标区累计至少 240 Tick，才查找阻挡者。
- 目标区驻留超过 720 Tick 后不再启动新的 Yield；这时让现有让路结束，并留给 900 Tick Overflow 硬兜底。
- 阻挡者必须已经 Arrived，或距离自身槽位不超过 18px；不能正在让路、Overflow 或冷却。
- 只考虑靠近“受阻单位当前位置 → 原目标”线段的阻挡者，并用 Unit ID 解决相同代价的确定性排序。
- 每 30 Tick 最多启动 2 次，全局最多 4 个活动 Yield，避免终点附近同时大幅重排。

### 3.2 临时让路点

候选方向由路径切线的左右两侧及背离编队中心的方向组成，并用两档距离和固定权重确定性采样。每个候选必须：

- 在世界边界内，且单位圆盘和到达线段均不穿过静态/动态障碍。
- 不与任何单位当前占用或普通槽位预留冲突。
- 不与其他 Yield 的原槽位返回预留或临时让路点冲突。

开始让路后，原槽位不会被释放：`SlotTarget`、`DestinationYieldReturnTarget` 和 `DestinationYieldPoint` 同时参与预留检查。因此后续命令波次和 Overflow 都不能抢占让路者的原位置或临时停靠位置。

### 3.3 等待、返回与有界失败

- `MovingAside` 最多持续 360 Tick；到达让路点后进入 `Waiting`。
- 当受助单位到达/接近其槽位、命令已改变、进入 Overflow，或等待期限到达时，让路者开始返回。
- `Returning` 最多持续 180 Tick；成功后清理状态并进入 600 Tick 冷却，避免同一单位反复开合通道。
- 如果返回仍不能完成，已经验证安全的让路点会提升为该单位的永久 Overflow；状态机仍会结束，不产生新的永久卡死。
- 新 Move 或 Stop 会清理旧 Yield 上下文，防止旧协作关系污染新命令。

## 4. 第三层：有预算的局部多单位重匹配

两单位换槽无法跳出三单位以上的循环局部最优。当前实现增加一个严格有界的局部 Hungarian 重匹配：

- 每 60 Tick 最多尝试一次，不持续运行全局匹配。
- 只处理至少 48 人、目标附近持续无进展、并且仍有足够移动单位的编队。
- 以确定性锚点选取最多 24 个局部候选，最多允许 4 个已就位阻挡者参与，避免把稳定外圈大规模拉走。
- 只交换当前已有槽位，不创建或释放预留；所有目标先原子写入，再统一进入路径预算。
- 至少 3 个单位改变槽位、总平方距离明确下降、每个单位退化不超过 18px，并且新直线路径可通行，计划才会提交。
- Hungarian 成本加入单位进入顺序与槽位深度的归一化秩差，提供轻量 SlotDepth 效果，但不维护全局连续深度场。
- 变更单位进入 180 Tick 冷却，避免局部多匹配与两单位换槽反复争夺同一组槽位。

本阶段最初没有加入完整碰撞挤压优先级；2026-07-14 的通用动态阻塞专项已经在独立策略层补齐，见第 8 节。局部重匹配仍只负责超车和三单位以上循环错位，不承担物理推挤。

## 5. 与其他系统的边界

- 新 Move 和 Stop 会清空全部终点 V2 历史。
- 普通局部换槽保留总驻留历史。
- 局部多单位重匹配同样保留总驻留历史，并与两单位换槽共用冷却。
- 动态路径重算和 `WaitingForPath` 不清空总驻留历史。
- Choke 控制期间不累计终点历史。
- Overflow 仍使用正常路径预算、Steering 和碰撞系统。
- Yielding 单位暂时退出局部换槽和 Overflow 候选集合，避免多个恢复器同时改写其目标。
- 后续 Move 会把 Overflow 槽位作为普通现有预留参与冲突检查。

## 6. 当前验收结果

### 临时编队阻挡

- 上一版：75/80 到达，5 个 Moving，最大误差 89.8px。
- 当前：80/80 到达，4 次 Yield，5 次 Overflow，结束时 0 个活动 Yield、0 Moving，最大误差 5.7px。

### 外圈封闭专用场景

测试先根据业务层可见的 `AssignedTarget` 选出 48 个最外层槽位，让它们先就位；随后释放 32 个内部槽位单位。

- 外圈：48/48 到达。
- 内部：32/32 到达。
- Yield：8。
- 结束时活动 Yield：0。
- Overflow：31；相比只有第一层时减少 1 次，表明已有内部单位穿过临时通道并回到原槽位体系。
- 最终 Moving：0。
- 重叠：0。
- 最大误差：7.2px。

### 其他改善

- 速度超车：80/80，1 次局部重匹配、原子重排 10 人，0 重叠，最大误差 7.7px。
- 角落混合半径：80/80，0 重叠，最大误差 3.5px。
- 普通密集编队：80/80，最大误差 4.9px。
- 192 单位：192/192，最大误差 3.7px。
- 动态群组改道：48/48，最大误差 6.1px。

## 7. A2 收口结论

当前 Demo 已形成完整的有限恢复链：

```text
初始唯一槽位
  → 两单位低成本换槽
  → 最多 24 人局部原子重匹配
  → 已就位阻挡者主动 Yielding
  → 唯一 Overflow 安全兜底
```

这已经满足当前 Demo 的 A2 目标：普通干扰、外圈封锁、超车重排、角落和混合半径都存在黑盒闭环，协作状态有界结束，1000 单位性能仍在预算内。

不继续在 A2 内追求完整碰撞挤压优先级、全局连续 SlotDepth、绝对最少 Overflow 或 SC2 级编队美学。它们作为未来实际战斗和操作体验暴露问题时的打磨项，不再阻塞下一阶段。下一阶段正式进入 Clearance 与 Movement Class。

## 8. R3 通用动态阻塞与优先级挤压专项（已完成，2026-07-14）

这次实现不只服务 Rally 或持续增援，而是位于全部地面导航共用的 Steering、圆碰撞和导航完成边界。最终机制只有两条主路径：

```text
仍可推进
  → 按 Movement Class 决定避让责任和穿透修正方向
  → 推动普通静止单位
  → 静止单位稳定后原子迁移全局 Reservation

确实无法推进
  → 错峰探测前方单位阻塞
  → 连续 180 Tick 阻塞且没有足够净进展
  → 当前导航段在当前位置完成
```

### 8.1 优先级合同

- 基础优先级直接使用现有 `Small / Medium / Large` Movement Class，不增加第二套质量配置。
- 高级移动单位可推动低级；同级移动单位可推动普通 Idle/Arrived 阻挡者；低级不能推动高级。
- 同级不能移动 Hold。活动战斗接敌同样退出普通行军强推，继续使用 `CombatContactPolicy` 的前摇、近战、冷却和移动武器阻力，避免破坏火力线或近战槽。
- Steering 只在候选速度真正产生重叠或 TTC 碰撞时计算让路责任；物理穿透阶段使用同一策略决定修正占比，避免“Steering 说我有路权，碰撞却把我弹回去”的矛盾。
- 被推动单位不会继续追逐旧落点。连续稳定 6 Tick 后才批量迁移 Reservation，避免每帧执行全局松弛；新 Reservation 保持至少 2px 目标净空，且距实际站位最多 12px。

### 8.2 三秒有界结束

- 只处理正常 `Moving`，不抢占 `PathPending`、Choke、采矿/施工碰撞豁免和正在 Yield 的状态机。
- 正常前进速度已足以在窗口内形成有效进展的单位不启动阻塞查询；其余前向查询使用 96px 预测走廊，因此单位在 TTC 避让阶段提前停住也能被识别，不要求已经发生圆盘穿透。
- 实际低速/停滞单位按 Unit ID 错峰，每 6 Tick 查询一次；确认阻塞后按真实 Tick 累计。180 Tick 窗口内若净进展超过 `max(12px, 1.5×radius)`，重新开始窗口。
- 超时后清路径、速度和移动组，把当前位置写成当前导航完成点。普通 Move 清除移动意图；没有活动战斗目标的 AttackMove 保留攻击语义并把恢复点收敛到当前位置；战术追击腿只结束当前追击导航，不覆盖原 AttackMove 槽，脱战后仍恢复原路线。Shift 队列仍可在同 Tick 的正式完成检查中进入下一项。
- Yield 冷却从永久禁用改为 600 Tick，单位经历一次成功让路后仍可参与更晚的新拥堵，但不会短期反复开合通道。

### 8.3 确定性与性能

`DynamicBlockageTicks`、窗口距离、推后 Reservation 迁移倒计时以及语义化移动目标与结果进入 Hot Snapshot v30 与 State Hash v31；非法负数、非有限距离和非法枚举在解码时拒绝。运行时使用预分配单位/目标/索引缓冲区，没有在每 Tick 推挤热点创建 `List` 或 `ToArray`。

同机 Release 时间门禁结果：256/512/1000 单位移动 P95 为 `1.85 / 5.42 / 16.18ms`，分别低于 `4 / 12.5 / 16.67ms` 预算；128/256 总单位持续战斗 P95 为 `0.82 / 2.66ms`，低于 `4 / 8ms` 预算。当前仓库的分配门禁在未修改 HEAD 对照上同样失败，属于此前功能合入后未更新的独立基线债务；本专项没有放宽分配预算，也不把该债务伪装成通过。

### 8.4 黑盒验收

- `dynamic-blockage-priority-matrix`：同级穿过、高推低、低级受高级阻挡、同级受 Hold 阻挡四条隔离车道；前两条阻挡者位移约 `537.5/538.3px`，后两条锚点漂移 `0.00px`，两次阻塞均有界结束，Tick 300 活跃态 Hot Restore 精确一致。
- `dynamic-blockage-continuous-waves`：`32 + 4 + 3 + 2 + 1` 五波、42 单位连续进入同一目标，`42/42` 到达、`42/42` 唯一 Reservation，稳定后 240 Tick 最大漂移 `0.00px`，且没有误触三秒兜底。
- `destination-outer-ring`：外圈 `48/48`、内圈 `32/32`，最终 `80/80` 到达、0 Moving、0 重叠。
- 历史交叉结果中上述场景均通过；其中 `combat-melee-slots` 已因错误成功合同退出活动门禁，其余项目仍保留原覆盖范围。
